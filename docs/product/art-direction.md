# Art Direction

## Главный визуальный референс
Главный референс по графике для OmniCorp:

```text
Streets of Rogue
```

Это не означает копирование ассетов, персонажей, UI или конкретных художественных решений. Это ориентир по общему ощущению и практичности визуального языка.

## Главный геймдизайн-ориентир

```text
OmniCorp = Streets of Rogue для AppSec
```

Расшифровка:

```text
киберпанковый Streets of Rogue, где вместо драк и краж - web-security миссии настоящими инструментами
```

Брать нужно не конкретный юмор, не спрайты и не визуальные детали, а структуру:

- маленький район;
- понятная цель;
- свобода подхода;
- быстрый результат;
- интерактивная среда;
- короткие миссии вместо длинных лекций.

## Что берем из референса
- Top-down / слегка псевдоизометрический игровой экран.
- Быстро читаемые персонажи и объекты.
- Компактные локации с высокой плотностью интерактивных элементов.
- Простая, но выразительная пиксельная/тайловая подача.
- Много маленьких визуальных подсказок на карте.
- Четкое различие зон, объектов, NPC и терминалов.
- Возможность быстро понимать, с чем можно взаимодействовать.

## Почему Streets of Rogue подходит

### 1. Top-down город
Игрок ходит по маленьким районам, заходит в здания, общается с NPC и выполняет задания.

Для OmniCorp это можно превратить в:
- район трущоб;
- офис OmniCorp;
- дата-центр;
- бар хакеров;
- полицейский участок;
- серверную;
- учебные лаборатории.

### 2. Маленькие миссии вместо простыней текста
В Streets of Rogue цель обычно простая и быстро читаемая.

Для OmniCorp это переводится в:
- получи чужой профиль;
- найди скрытый endpoint;
- перехвати запрос;
- обойди проверку роли;
- докажи XSS;
- найди токен;
- выбери безопасную конфигурацию.

### 3. Системность
Мир должен реагировать на действия игрока, но в рамках реалистичного scope для маленькой команды.

Примеры для OmniCorp:
- охранник блокирует вход;
- терминал требует бейдж;
- NPC дает подсказку;
- камера видит игрока;
- серверная закрыта до выполнения миссии;
- доступ к зоне открывается после proof/flag.

### 4. Функциональная графика
Streets of Rogue не выглядит как дорогая пиксельная картина. Он функциональный, быстрый и читаемый.

Для OmniCorp это важно, потому что игра должна быть:
- понятной;
- быстрой в производстве;
- расширяемой;
- пригодной для большого количества миссий.

## Роли разных референсов

### Streets of Rogue
Главный референс по:
- структуре уровней;
- интерактивности;
- компактным районам;
- свободе подхода;
- коротким миссиям.

### Stardew Valley
Референс по:
- читаемости 2D-графики;
- простоте персонажей;
- понятности тайлов;
- мягкой визуальной доступности.

### Hacknet
Референс по:
- атмосфере хакерской игры;
- напряжению;
- ощущению работы через терминал.

Не использовать Hacknet как референс по инструментам, потому что OmniCorp должен опираться на реальные web security инструменты.

### VA-11 Hall-A
Референс по:
- киберпанк-настроению;
- диалоговой атмосфере;
- ощущению жизни внутри города.

### PortSwigger Academy
Референс по:
- реальности уязвимостей;
- качеству обучения;
- проверке результата;
- практическим web security сценариям.

### Burp Suite, OWASP ZAP, curl
Референс по:
- реальному workflow;
- синтаксису;
- инструментам;
- способу решения миссий.

## Что важно для OmniCorp
OmniCorp - это не просто экшен-игра, а тренажер безопасной разработки. Поэтому графика должна помогать обучению:

- терминалы должны быть заметны;
- интерактивные объекты должны читаться сразу;
- зоны миссий должны визуально отличаться;
- UI не должен мешать карте;
- текстовые подсказки должны быть короткими;
- результат миссии должен быть понятен без лишнего шума.

## Визуальная формула

```text
Streets of Rogue structure + Stardew Valley readability + VA-11 Hall-A mood + PortSwigger realism + real AppSec tools
```

## Не делать
- Не уходить в реалистичную графику.
- Не делать темную кашу без читаемых объектов.
- Не копировать конкретный стиль, ассеты или персонажей Streets of Rogue.
- Не перегружать карту декором, который мешает миссиям.
- Не делать UI похожим на фейковый "hack simulator".

## Практический ориентир для прототипа
Ближайшие визуальные шаги:

1. Сделать тайловую учебную комнату.
2. Заменить текущий силуэт игрока на пиксельного персонажа.
3. Сделать терминалы как понятные интерактивные объекты.
4. Добавить подсветку объектов при приближении.
5. Разделить комнаты по темам: HTTP, cookies, auth, API.
6. Позже добавить NPC и сюжетные зоны.
